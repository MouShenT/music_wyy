## 一、登录方案：Cookie 登录（唯一方式）

**只用 Cookie，不需要账号密码。**

| 步骤 | 说明 |
|------|------|
| 1. 获取 Cookie | 用户从浏览器开发者工具或抓包获取完整 Cookie 字符串 |
| 2. 提取 MUSIC_U | Cookie 中的核心字段 `MUSIC_U`（令牌），其他字段可选 |
| 3. 验证登录态 | 调用 `/login/status` 传入完整 Cookie 或 `cookie` 参数，返回用户信息即登录成功 |
| 4. 刷新登录态 | Cookie 有效期约 30 天，过期后调用 `/login/refresh` 刷新，或提示用户重新获取 |

**API 端点**:
- `/login/status` — 验证 Cookie 并获取用户信息
- `/login/refresh` — 刷新过期 Cookie

**安全要点**:
- Cookie 只在本地客户端存储，不上传服务器
- 不做中间人转发（所有 API 请求直接从客户端 → NeteaseCloudMusicApi 后端）
- 可选：支持多账号 Cookie 切换

**不使用的登录方式**（明确排除）:
- ~~手机号+密码~~ → 需要处理密码加密、验证码，安全风险高
- ~~扫码登录~~ → 交互复杂，不适合工具类 App
- ~~手机验证码~~ → 依赖第三方短信接口

## 二、五大核心功能模块

### 模块 1：批量歌单管理（P0 — 最高优先级）

**用户痛点**:
- 官方客户端仅支持单首操作，整理 200 首歌单平均耗时 47 分钟
- 无 Ctrl/Shift 多选、无右键批量操作
- API 歌单 1000 首硬限制，无官方分页

**功能清单**:

| 功能 | 说明 | API |
|------|------|-----|
| **批量创建歌单** | 一键按模板批量创建多个歌单（如按语种/年代/风格） | `/playlist/create` |
| **批量添加歌曲** | 文本列表/Excel 导入 → 搜索匹配 → 批量添加到指定歌单 | `/cloudsearch` + `/playlist/tracks` (op=add) |
| **批量删除歌曲** | 多选后一键从歌单中移除 | `/playlist/track/delete` |
| **歌单合并** | N 个歌单合并为一个，自动去重 | `/playlist/track/all` + `/playlist/tracks` |
| **歌单拆分** | 按歌手/语种/年代/风格自动拆分为多个歌单 | `/playlist/create` + `/playlist/tracks` |
| **歌单排序** | 按发行时间/播放量/歌手/专辑名排序（需全量获取后本地排序再写回） | `/playlist/track/all` |
| **批量去重** | 扫描歌单内重复歌曲，保留其一或全部删除 | `/playlist/track/all` |
| **歌单对比** | 两个歌单差异对比（A 有 B 无 / B 有 A 无） | `/playlist/track/all` |
| **大歌单穿透 1000 首限制** | 先拿 trackIds，再分批 `song/detail`（每批 500 首） | `/playlist/detail` + `/song/detail` |

**实现关键点**:
- 超过 1000 首的歌单：先通过 `/playlist/detail` 获取 trackIds 列表，再用 `/song/detail?ids=id1,id2,...` 分批获取歌曲详情（每批上限约 500 首）
- 批量添加时需逐首搜索匹配，请求间隔 ≥300ms 防止 429
- 使用 `/batch` 端点可将多个请求合并为一次 HTTP 调用

### 模块 2：自动化签到与任务（P0 — 高价值零成本）

**用户痛点**:
- 手动签到易遗忘，连续 365 天断签即归零
- 冲 LV10 需累计 10 万首播放，手动几乎不可能
- 云贝/音乐人任务每天需逐一手动点击
- 第三方脚本 API 频繁变更导致断签后才发现

**功能清单**:

| 功能 | 说明 | API |
|------|------|-----|
| **每日签到** | Web 端 + Android 端双平台签到 | `/daily_signin` |
| **云贝签到** | 自动完成每日云贝任务 | `/yunbei/sign` + `/yunbei/tasks` 系列 |
| **刷歌打卡** | 自动播放指定歌单刷听歌量（每日上限约 300 首） | `/scrobble` |
| **VIP 成长值** | 自动领取会员任务成长值 | `/vip_growthpoint` |
| **音乐人任务** | 音乐人签到、发布动态、回复评论/私信 | `/musician/sign` 等 |
| **签到结果推送** | 通过 Server酱/WxPusher/PushPlus/Bark 推送每日签到结果 | — |

**风控规避策略**:
- 单次请求间隔 ≥300ms（建议 500ms+）
- 每日 scrobble 总量控制在 300 首以内
- 不要频繁登录登出（容易触发账号异常）
- 刷歌时优先使用未播放过的歌曲（已听过的歌曲可能不计入播放量）
- 每天打卡 2-3 次为上限

**架构建议**: 支持云函数/青龙面板/Docker 定时执行

### 模块 3：歌单数据统计与分析（P1）

**用户痛点**:
- 官方「红心歌单智能筛选」仅覆盖最近 100 首
- 无全量听歌数据分析
- 年度报告数据不准（2025 年末大规模吐槽）

**功能清单**:

| 功能 | 说明 |
|------|------|
| **全量歌单分析** | 分析所有歌单（非仅 100 首），覆盖全量歌曲 |
| **歌手分布饼图** | 统计歌单中各歌手占比 |
| **年代/语种/风格分布** | 按发行年份、语言、流派分类统计 |
| **播放热度排行** | 按播放量/收藏量排序歌曲 |
| **月度/年度听歌报告** | 自定义样式的个人听歌报告 |
| **重复歌曲报告** | 跨歌单检测重复歌曲 |
| **歌曲下架/变灰检测** | 扫描所有歌单，标记已变灰（不可用）的歌曲 |

**实现关键点**:
- 需先获取全量歌曲数据（处理 1000 首限制）
- 歌曲元数据（发行时间、语种等）来自 `/song/detail` 返回的字段
- 播放次数来自 `/user_record`（听歌排行）

### 模块 4：跨平台歌单迁移（P1 — 用户刚需）

**用户痛点**:
- 从 Spotify/QQ音乐/Apple Music 迁移到网易云时，只能逐首搜索添加
- 平台间无官方迁移工具
- 歌曲匹配不准确（仅靠歌名）

**功能清单**:

| 迁移方向 | 说明 |
|----------|------|
| **导入 → 网易云** | 从 Spotify 歌单链接 / QQ音乐歌单链接 / 文本歌单列表 导入 |
| **网易云 → 导出** | 导出为 JSON / Excel / CSV / Spotify 格式 / 纯文本列表 |
| **智能匹配** | 歌曲名 + 歌手名双重匹配，提高准确率 |
| **匹配报告** | 标注成功匹配 / 未匹配 / 部分匹配的歌曲 |
| **一键创建** | 匹配完成后在网易云一键创建对应歌单 |

**导入流程**:
```
用户输入外部歌单（链接/文本）
    → 解析歌曲列表（歌名 + 歌手）
    → 网易云 API 逐首搜索（/cloudsearch，歌曲名+歌手名）
    → 匹配度评分（完全匹配 / 部分匹配 / 未找到）
    → 用户确认 → 创建歌单 + 批量添加
```

### 模块 5：监控与告警（P2）

**用户痛点**:
- 歌单中歌曲悄悄变灰，用户发现时已无法收听
- 会员到期导致已下载歌曲失效
- 没有主动通知机制

**功能清单**:

| 功能 | 说明 |
|------|------|
| **歌曲灰掉监控** | 定时扫描歌单，检测歌曲可用性变化，标记变灰歌曲 |
| **版权到期预警** | 检测即将下架的歌曲（通过官方版权到期信息） |
| **会员到期提醒** | VIP 到期前 3/7/15 天推送提醒 |
| **歌单变化监控** | 关注的歌单有新增/删除歌曲时通知 |
| **收藏歌手新歌提醒** | 关注歌手发布新歌/新专辑时推送 |

## 四、登录态管理设计

```
┌──────────────────────────────────────────┐
│  用户输入 Cookie（粘贴 MUSIC_U）           │
└──────────────┬───────────────────────────┘
               │
┌──────────────▼───────────────────────────┐
│  调用 /login/status 验证                  │
│  ├─ 成功 → 解析用户信息，存储 Cookie      │
│  │        Cookie 只在本地存储（SharedPref /│
│  │        Keychain / localStorage）       │
│  └─ 失败 → 提示 Cookie 无效，请重新获取    │
└──────────────┬───────────────────────────┘
               │
┌──────────────▼───────────────────────────┐
│  定时检查 /login/status                   │
│  ├─ 有效 → 继续                           │
│  └─ 过期 → 静默调用 /login/refresh 刷新   │
│            ├─ 成功 → 更新 Cookie           │
│            └─ 失败 → 通知用户重新登录      │
└──────────────────────────────────────────┘
```

## 五、关键约束与注意

| 约束 | 说明 |
|------|------|
| 请求频率 | 每两次请求间隔 ≥ 300ms（推荐 500ms），否则触发 429 限流 |
| 歌单上限 | 单歌单 1000 首 API 硬限制，需变通方案处理 |
| 音质限制 | 付费歌曲只能拿到 128k，无损/Hi-Res 不可获取 |
| 版权限制 | 灰色歌曲无法获取播放 URL |
| Cookie 有效期 | 约 30 天，过期需刷新或重新获取 |
| `/batch` 端点 | 可将多个请求合并为一个 HTTP 调用，减少请求次数 |

## 六、MVP 分期规划

### 第一期（P0 — 核心闭环）
- [ ] Cookie 登录（支持多账号切换）
- [ ] 批量导入歌单（文本列表 → 搜索匹配 → 创建歌单 → 添加歌曲）
- [ ] 歌单导出（JSON / Excel / 纯文本）
- [ ] 每日自动签到 + 推送通知

### 第二期（P1 — 管理增强）
- [ ] 批量去重 / 歌单合并 / 歌单拆分
- [ ] 全量歌单数据分析（歌手/年代/语种分布）
- [ ] 跨平台导入（Spotify / QQ音乐 → 网易云）
- [ ] 歌曲变灰检测

### 第三期（P2 — 智能体验）
- [ ] AI 歌单生成（LLM 根据心情/场景推荐歌曲列表）
- [ ] 收藏歌手新歌监控告警
- [ ] 会员到期提醒
- [ ] 月度听歌报告自动生成
